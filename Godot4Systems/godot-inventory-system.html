📦 INVENTORY SYSTEM

Drag & Drop Inventory
for Godot 4

4-file drop-in inventory system. ItemDB autoload, typed ItemData resources, PlayerInventory logic, and a full drag & drop UI — ready in minutes.

Godot 4.x GDScript RPG Survival Any Genre No Plugins Mobile Ready

1. What's Included

ItemData.gd
RESOURCE · CLASS

Typed Resource class for all item definitions. Supports guns, ammo, meds, melee, nades, helmets, vests, bags. Set everything in the Godot Inspector — no code needed to define items.

ItemDB.gd
AUTOLOAD · SINGLETON

Scans res://Resources/ at startup and builds a global item dictionary. Call ItemDB.get_item("id") from anywhere.

PlayerInventory.gd
NODE · CORE LOGIC

All inventory state and logic. Equipment slots, item stacking, mag ammo persistence, weapon cycling, query helpers. Emits signals on every change — no polling needed.

Inventory.gd + .tscn
CONTROL · UI

Full drag & drop UI. Touch + mouse support. Equipment panels, item grid, med use with progress overlay, animated slots. Reference implementation — replace or extend freely.

ℹ️
UI is optional. ItemData, ItemDB, and PlayerInventory are fully self-contained backend logic. Build your own UI on top, or use the included scene as a starting point.

2. Setup — 4 Steps

  1. Register ItemDB as AutoLoad Project → Project Settings → AutoLoad → Add res://ItemDB.gd with name ItemDB
  2. Create your item .tres files Right-click in FileSystem → New Resource → pick ItemData → fill in Inspector → save to res://Resources/
  3. Add PlayerInventory to your player scene Add a PlayerInventory node as a child of your player. That's it — no exports required.
  4. Instance the Inventory scene (optional UI) Add Inventory.tscn to your HUD. Assign the inventory export to your PlayerInventory node. Call inventory_ui.open() / .close().
✅
ItemDB is zero-config. Just drop your .tres files in res://Resources/ and they're available everywhere instantly.

3. ItemData — Fields Reference

All fields are @export — configure everything in the Godot Inspector. Only fill in the group relevant to your item type.

📄 ItemData.gd — field groups
# ── Shared (every item) ──────────────────────────────────
@export var id          : String    # unique key, e.g. "ammo_ar"
@export var name        : String    # display name
@export var item_type   : ItemType  # GUN / AMMO / MED / MELEE / NADE / ...
@export var rarity      : Rarity    # COMMON / UNCOMMON / RARE / LEGENDARY
@export var icon        : Texture2D # shown in inventory slots
@export var scene_path  : String    # 3D model path (optional)

# ── Gun ──────────────────────────────────────────────────
@export var gun_type    : GunType   # AR / SMG / SNIPER
@export var ammo_type   : String    # matches ammo item id
@export var damage      : int
@export var fire_rate   : float
@export var mag_size    : int
@export var reload_time : float

# ── Med ──────────────────────────────────────────────────
@export var health_amount : int
@export var is_overtime   : bool   # true = heals over time (drink)
@export var heal_duration : float  # seconds, used if is_overtime = true

# ── Melee ────────────────────────────────────────────────
@export var melee_damage    : int
@export var melee_range     : float
@export var melee_swing_time: float

# ── Nade ─────────────────────────────────────────────────
@export var nade_damage       : int
@export var nade_blast_radius : float
@export var nade_fuse_time    : float
@export var nade_throw_force  : float
⚠️
id must be unique. ItemDB keys by id. If two .tres files share the same id, the second one silently overwrites the first.

5. Code Examples

Add items when player picks up loot

📄 pickup.gd
extends Area3D

@export var item_id  : String = "ammo_ar"
@export var quantity : int    = 30

func _on_body_entered(body: Node3D) -> void:
    var inv = body.get_node_or_null("PlayerInventory")
    if not inv: return
    inv.add_item(item_id, quantity)
    queue_free()

Listen to inventory signals in your player

📄 player.gd
@onready var inventory : PlayerInventory = $PlayerInventory

func _ready() -> void:
    inventory.gun_equipped.connect(_on_gun_equipped)
    inventory.melee_equipped.connect(_on_melee_equipped)
    inventory.mode_changed.connect(_on_mode_changed)
    inventory.inventory_updated.connect(_on_inventory_changed)

func _on_gun_equipped(slot: int, gun: ItemData) -> void:
    print("Equipped: ", gun.name, " in slot ", slot)
    # Spawn 3D gun model, update HUD, etc.

func _on_mode_changed(mode: PlayerInventory.WeaponMode) -> void:
    match mode:
        PlayerInventory.WeaponMode.GUN:   _show_gun_anim()
        PlayerInventory.WeaponMode.MELEE: _show_melee_anim()
        PlayerInventory.WeaponMode.NADE:  _show_nade_anim()

Query helpers — drive player logic flags

📄 player.gd — combat flags
func _update_combat_flags() -> void:
    has_weapon   = inventory.has_any_gun() or inventory.has_melee()
    has_ammo     = inventory.has_any_ammo()
    can_throw    = inventory.has_nade()
    can_heal     = inventory.has_medkit() or inventory.has_drink()

func _input(event: InputEvent) -> void:
    if Input.is_action_just_pressed("cycle_gun"):
        inventory.cycle_active_gun()   # Q key
    if Input.is_action_just_pressed("melee_mode"):
        inventory.set_mode(PlayerInventory.WeaponMode.MELEE)

6. Scene Tree — Inventory.tscn

The included UI scene expects this exact node structure. Node names are case-sensitive.

Inventory (Control) ← Inventory.gd attached here └── Panel └── HBoxContainer ├── assets ← equipment slots │ ├── gun1 (Panel) │ │ └── VBoxContainer │ │ ├── TextureRect ← gun icon │ │ └── gunname (Label) │ ├── melee (Panel) │ │ └── VBoxContainer │ │ ├── TextureRect │ │ └── gunname (Label) │ └── nade (Panel) │ └── VBoxContainer │ ├── TextureRect │ └── gunname (Label) └── items ← item grid └── GridContainer └── in1 (Panel) ← slot template (visible=false) └── VBoxContainer ├── HBoxContainer │ ├── TextureRect ← item icon │ └── Label ← stack qty └── name (Label) ← item name
ℹ️
in1 is duplicated at runtime. The slot template is hidden and cloned for each item. Don't delete it — the system won't work without it.

8. FAQ

Can I use this for an RPG, not a shooter?

Yes. ItemData supports any type. Remove or ignore the gun/ammo groups. The stacking, drag & drop, and signal system works for any genre. The WeaponMode enum is also fully optional.

Where do I put my .tres item files?

Exactly res://Resources/ — ItemDB scans this folder on startup. Subfolders are not scanned. All files must end in .tres.

Does it support multiplayer?

The backend logic (PlayerInventory, ItemDB) is multiplayer-agnostic. The included UI has an is_local flag — call init(true/false) to show/hide it per player. Syncing inventory state over the network is up to you.

Does it work on mobile / touch screens?

Yes. The drag system handles both InputEventScreenTouch and InputEventScreenDrag natively. Tested on Android.

Can I replace the UI with my own?

Absolutely — that's the point. Inventory.gd + .tscn are a reference implementation. The backend (ItemData, ItemDB, PlayerInventory) has no UI dependency at all.
