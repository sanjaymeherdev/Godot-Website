📦 1. What is a Keystore?

A keystore is a digital signature file that proves you are the author of your app. Every Android app must be signed before it can be distributed — the keystore is what you use to sign it.

Think of it like a wax seal on a letter. Anyone can read the letter, but only you have the seal. When you update the app later, Android checks that the new version has the exact same seal before allowing the update. No seal match — no update.

🔏 Digital Signature

Cryptographically proves the APK or AAB was built by you and hasn't been tampered with.

🔄 Required for Updates

You must use the exact same keystore to sign every future update of your app. No exceptions.

🗄️ Contains Two Things

The keystore file (.keystore or .jks) and the alias inside it — each protected by their own password.

⛔ If You Lose Your Keystore

  • You cannot update your existing app on Play Store, Amazon, or anywhere
  • You must publish as a completely new app with a new package name
  • All existing users, reviews, and ratings are gone
  • Google cannot recover it for you — nobody can

📁 Two Types of Keystore

Debug keystore — auto-created by Godot/Android tools for testing. Different on every machine. Never use for release.

Release keystore — you create this once and keep it forever. This is what goes on the Play Store.

❓ 2. Do You Need a Keystore?

Scenario Need Release Keystore? Why
Testing on your own phone (USB) NO Debug keystore auto-created by Godot
Sharing APK with a friend directly OPTIONAL Works either way, but release is cleaner
Firebase / Google Sign-In integration YES SHA fingerprint required from your keystore
AdMob setup YES Signed release APK required for submission
Amazon App Store YES Requires signed release APK
Google Play Store YES Mandatory — upload key signs your AAB
itch.io (Android APK) RECOMMENDED Not required but professional practice

✅ Rule of Thumb

If anyone other than you is running your app, sign it with your release keystore. Always.

🛠️ 3. Generate Your Keystore

Create your keystore once and never again for the same app. Pick any method below — the result is identical.

Method 1: Godot Editor Method 2: Command Line Method 3: Android Studio

1 Godot Editor — Easiest, No Extra Tools

No Java, no terminal. Built directly into Godot 4.

Steps inside Godot 4 Editor: 1. Open your project 2. Go to Project → Export... 3. Add or select the Android preset 4. Scroll down to Keystore → Release section 5. Click the folder icon next to the path field 6. In the file dialog, click "New Keystore" (bottom area) 7. Fill in the form below 8. Save to a location you will NOT lose (NOT inside your project folder)

📋 Keystore Fields — Fill These In:

Strong password. You need this every time you export a release build. Save it.
Usually your game or app name. No spaces. Lowercase recommended.
Can be the same as keystore password for simplicity.
Google recommends 25+ years. Play Store requires validity past Oct 2033.
Two-letter ISO code. IN for India.

⚠️ Do NOT save the keystore inside your project folder

If you push your project to GitHub or share it, the keystore must not be included. Save it somewhere separate — a dedicated folder on another drive or cloud storage.

2 Command Line — keytool (comes with Java JDK / Android Studio)

If you have Android Studio installed, you already have keytool.

$ keytool -genkey -v \ -keystore mygame.keystore \ -alias mygame_key \ -keyalg RSA \ -keysize 2048 \ -validity 9125 # 9125 days = 25 years # Run this in the folder where you want the .keystore file saved Enter keystore password: [type your password] Re-enter new password: [type again] What is your first and last name? Sanjay Meher What is your organizational unit? Development What is your organization? Champ Gaming What is your City or Locality? Delhi What is your State or Province? Delhi Two-letter country code: IN Is CN=Sanjay Meher, O=Champ Gaming, C=IN correct? [no]: yes ✓ Keystore created: mygame.keystore

✅ Where is the file?

In whatever folder your terminal was pointing to when you ran the command. Run pwd (Mac/Linux) or cd (Windows) to see the current folder.

3 Android Studio — GUI Method

If you already have Android Studio open for other reasons.

Steps in Android Studio: 1. Open any Android project (even a blank one) 2. Go to Build → Generate Signed Bundle / APK 3. Select APK → Next 4. Click "Create new..." under Key store path 5. Fill in the same fields as Method 1 6. Click OK — file is saved where you specified # You don't actually need to finish building an APK. # Just creating the keystore file is enough. Cancel after step 6.

🔑 4. SHA Fingerprints

What is a SHA Fingerprint?

A SHA fingerprint is a unique identifier derived from your keystore. It's a long string of hex characters that represents your signing key mathematically.

Services like Firebase, AdMob, and Google Sign-In use it to verify that the APK talking to their servers was actually signed by you — not someone copying your app ID.

SHA-1

Older format. Still required by:

  • Firebase (development)
  • Google Maps SDK
  • Google Sign-In (older flow)
  • Some older SDKs

⚠️ SHA-1 alone is not secure. Always add SHA-256 too.

SHA-256

Modern format. Required by:

  • Firebase (production)
  • Google Sign-In (modern)
  • Play Integrity API
  • AdMob app verification
  • Most new Google services

✅ Always register both SHA-1 and SHA-256.

MD5

Very old format. Rarely needed today. Some older tools ask for it.

❌ Never use MD5 for security purposes.

The keytool command gives you all three at once anyway.

1 Get SHA from Your Release Keystore

Run this command pointing at your .keystore file:

$ keytool -list -v \ -keystore mygame.keystore \ -alias mygame_key # Replace mygame.keystore with your actual file path # Replace mygame_key with your actual alias name # It will ask for your keystore password Alias name: mygame_key Creation date: ... Entry type: PrivateKeyEntry Certificate fingerprints: MD5: AA:BB:CC:DD:... SHA1: 12:34:56:78:9A:BC:DE:F0:12:34:56:78:9A:BC:DE:F0:12:34:56:78 SHA256: AB:CD:EF:01:23:45:67:89:AB:CD:EF:01:23:45:67:89:AB:CD:EF:01:23:45:67:89:AB:CD:EF:01:23:45:67

⚠️ Colons vs No Colons

The output shows SHA with colons (AB:CD:EF:...). Some services want it without colons (ABCDEF...). Firebase accepts both. AdMob usually wants no colons. Check what the service requires.

2 Get SHA from the Debug Keystore

Needed when testing Firebase or Google Sign-In before your release build is ready. The debug keystore is auto-created by Android tools and lives here:

# Debug keystore location by OS: Windows: C:\Users\YourUsername\.android\debug.keystore Mac / Linux: ~/.android/debug.keystore
# Windows command: > keytool -list -v -keystore "%USERPROFILE%\.android\debug.keystore" -alias androiddebugkey -storepass android -keypass android # Mac / Linux command: $ keytool -list -v -keystore ~/.android/debug.keystore -alias androiddebugkey -storepass android -keypass android # The debug keystore always uses: # alias: androiddebugkey # password: android # No need to type a password — it's hardcoded

⛔ Debug SHA is Different on Every Machine

The debug keystore is generated fresh on each computer. If you switch machines or reinstall Android Studio, your debug SHA changes. Always register both debug SHA (for development) and release SHA (for production) in Firebase / Google Console.

3 Where to Register Your SHA Fingerprints

Service SHA Needed Where to Add It
Firebase SHA-1 + SHA-256 Firebase Console → Project Settings → Your App → Add fingerprint
Google Sign-In SHA-1 (+ SHA-256 for modern) Google Cloud Console → APIs & Services → Credentials → OAuth client
Google Maps SDK SHA-1 Google Cloud Console → API Key → Android restrictions
AdMob SHA-256 (of signed APK) AdMob Console → App → App settings → App signing
Play Console Auto-managed Play Console shows your SHA automatically under Setup → App signing

💡 Play Console SHA — Important Detail

When Play App Signing is enabled (default for new apps), the SHA shown in Play Console is for Google's app signing key — NOT your upload keystore. This is the SHA you add to Firebase for production. Your upload keystore SHA is a separate entry for development. Add both. More on this in Section 6.

⚙️ 5. Configure Godot Export Settings

Godot 4 → Project → Export → Android preset: Keystore / Release section: Release: [path to your .keystore file] Release User: mygame_key ← your alias name Release Password: YourPassword ← keystore password Keystore / Debug section (leave defaults for testing): Debug: [leave blank — Godot auto-fills debug keystore] Gradle Build: Use Gradle Build: ✓ Enabled ← Required for AAB and some plugins Permissions (add what your game needs): INTERNET: ✓ ← Required for multiplayer, ads, analytics ACCESS_NETWORK_STATE: ✓ ← Required for AdMob VIBRATE: ✓ ← For haptic feedback RECORD_AUDIO: ✓ ← Only if voice chat or mic input used

🐛 Debug Export

Uses the auto-generated debug keystore. For testing on your phone only. Do not share or submit.

🚀 Release Export

Uses your keystore. This is what you submit everywhere — Play Store, AdMob, Amazon. Always use release for distribution.

⚠️ Gradle Build Required for AAB

If you're exporting an AAB for Play Store, Use Gradle Build must be enabled in the Android export preset. Without it, the AAB export option won't appear. Gradle Build also required for Admob plugin, Firebase plugin, and most Android-specific plugins.

💡 Godot Stores Your Keystore Path in the .export_credentials File

Godot saves keystore paths and passwords in export_credentials.cfg in your project root. This file is auto-added to .gitignore by Godot but double-check it is excluded before pushing to GitHub. Never commit this file.

🔐 6. Play App Signing — Upload Key vs App Signing Key

This confuses almost everyone. Here's what's actually happening.

Since 2021, Google Play uses Play App Signing by default for all new apps. This means Google holds the final signing key, and you only use your keystore as an upload key.

You sign AAB
with your Upload Key
→
Upload to
Play Console
→
Google re-signs
with App Signing Key
→
Delivered to users
signed by Google's key

Upload Key (Your Keystore)

  • You create and manage this
  • Only used to upload to Play Console
  • If lost — you can request a key reset from Google (one time)
  • SHA = your keystore SHA

App Signing Key (Google's Key)

  • Google creates and holds this
  • Used to sign the actual APK delivered to users
  • Cannot be changed or lost by you
  • SHA = shown in Play Console → Setup → App signing

⚠️ Which SHA to Add to Firebase?

For development/testing — add your upload key SHA (from your .keystore file).

For production — add the App Signing Key SHA shown in Play Console → Setup → App signing → App signing key certificate.

Add both to Firebase. Otherwise Firebase auth works in testing but breaks for Play Store users.

✅ The Good News About Play App Signing

Because Google holds the app signing key, losing your upload keystore is not catastrophic anymore. You can request a key reset from Google Play support. You get one reset per app lifetime. Treat your upload keystore like it matters — but know there's a recovery path unlike the old days.

Where to find the App Signing Key SHA in Play Console: Play Console → Your App → Setup → App signing → App signing key certificate → SHA-1 fingerprint ← copy this → SHA-256 fingerprint ← copy this too # These are the SHAs to register in Firebase for production # Your upload key SHA goes in Firebase separately for development

📦 7. AAB vs APK — Which to Use and When

Format Full Name Use For Signed By
APK Android Package Direct install, Amazon, itch.io, sideloading, AdMob setup Your keystore directly
AAB Android App Bundle Google Play Store (required since 2021) Your keystore → Google re-signs

APK — Use When:

  • Submitting to AdMob for fingerprint verification
  • Uploading to Amazon App Store
  • Sharing directly with testers
  • Publishing on itch.io
  • Internal testing / QA

AAB — Use When:

  • Uploading to Google Play Store
  • Required since Aug 2021 for new apps
  • Google optimises it per device (smaller download)
  • Requires Gradle Build enabled in Godot

💡 AAB Cannot Be Installed Directly

An AAB is not a complete installable file. It's a publishing format that Play Store converts to device-specific APKs. You cannot sideload an AAB. If you need to test the exact Play Store build, use Play Console's Internal Testing track to distribute to your own device.

Godot 4 — Export as AAB: Project → Export → Android preset → Export Project (bottom right) → Filename: mygame.aab → Make sure Export AAB checkbox is ticked → Requires Use Gradle Build to be enabled Godot 4 — Export as APK: Same as above, but → Filename: mygame.apk → Uncheck Export AAB (or just save as .apk) # Always use Release configuration for both # Debug builds are for testing on your own device only

📛 8. Package Name & Keystore — The Connection

Your keystore and package name are a pair. Break either one and updates break.

Android identifies your app by two things together: the package name (com.yourname.yourgame) and the signing keystore. Both must match across every update you ever publish.

Package Name Rules

  • Set in Godot → Project Settings → Application → Config → Name (and the Android package field)
  • Format: com.studio.gamename
  • All lowercase, no spaces, no hyphens
  • Must be globally unique on Play Store
  • Cannot be changed after publishing

What Happens if You Change Either

  • Change package name → treated as a completely new app
  • Change keystore → update rejected, cannot replace existing app
  • Change both → new app, old one still exists separately
  • Existing users get no update — they keep old version forever
Where to set Package Name in Godot 4: Project Settings → Application → Config: Name: My Game Version: 1.0.0 Export → Android preset → Options: Package / Unique Name: com.champgaming.mygame ^ this is your package name Version Code: 1 ← increment every upload Version Name: 1.0.0 # Version Code must increase with every upload to Play Store # Version Name is what users see ("1.0.0", "1.2", etc.) # Package Unique Name must NEVER change after first publish

⛔ Set Your Package Name Before First Publish

Decide on com.yourstudio.yourgame once and lock it in. After you publish to Play Store even once, it cannot be changed. Ever.

💾 9. Backup Strategy

⛔ This is the Most Important Section in This Guide

Everything else can be redone. The keystore cannot. Treat it like your passport.

📁 What to Back Up

  • The .keystore file itself
  • Keystore password
  • Alias name
  • Alias password
  • The package name tied to it
  • Date created

📍 Where to Store

  • Google Drive (personal, not shared)
  • Dropbox or OneDrive
  • External hard drive
  • USB stick (stored separately)
  • Password manager (Bitwarden, 1Password) for credentials

❌ Never Store Here

  • Inside your Git repository
  • In your project folder
  • On a shared drive others access
  • Only on your local machine (one copy = no backup)
  • Anywhere public
===== KEYSTORE BACKUP — KEEP PRIVATE ===== Date Created: 2025-04-20 App Name: My Game Name Package Name: com.champgaming.mygame Keystore File: mygame.keystore Keystore Password: YourPasswordHere Alias Name: mygame_key Alias Password: YourPasswordHere Validity: 25 years (until 2050) Created By: Sanjay Meher Organization: Champ Gaming Country: IN --- BACKUP LOCATIONS --- 1. Google Drive: /Keystores/mygame/ 2. External HDD: /GameDev/Keystores/ 3. USB: /Backups/Keystores/ --- SHA FINGERPRINTS --- SHA-1: XX:XX:XX:XX:... (run keytool -list -v to get) SHA-256: XX:XX:XX:XX:... Play Console App Signing Key SHA-256: XX:XX:XX:XX:... (from Play Console → Setup → App signing) ==========================================

✅ Backup Rule: 3-2-1

3 copies of the file. 2 different storage types (cloud + physical). 1 copy off-site or on a separate account. If this is important to you, treat it like that.

✅ 10. Pre-Launch Checklist

Tap each item to check it off before submitting to any store.

🔐 Keystore Setup

  • Release keystore created (not debug)
  • Keystore saved outside the project folder
  • Keystore backed up in at least 2 locations
  • Keystore password saved in a password manager
  • Alias name and alias password also saved
  • SHA-1 and SHA-256 fingerprints written down
  • export_credentials.cfg is in .gitignore

📦 Export Settings

  • Release keystore path set in Godot export preset
  • Alias name and password entered correctly
  • Package name set and confirmed (cannot change later)
  • Version Code set to 1 (increment for every upload)
  • Gradle Build enabled (required for AAB and plugins)
  • Correct permissions added (INTERNET, etc.)
  • Exported as Release (not Debug)

🔑 SHA & Services

  • SHA-1 + SHA-256 added to Firebase (if using Firebase)
  • Both debug SHA and release keystore SHA added to Firebase
  • Play Console App Signing Key SHA added to Firebase (for production)
  • Signed release APK tested on a real device
  • AdMob signed APK submitted for verification (if using AdMob)

🚀 Play Store Upload

  • AAB exported (not APK) for Play Store
  • Play App Signing accepted in Play Console
  • App signing key SHA noted from Play Console for Firebase
  • Version Code incremented from any previous upload

❓ 11. FAQ

Can I use the same keystore for multiple apps? ▼
Technically yes, but not recommended. Using one keystore for all your apps means if it's compromised, all apps are compromised. Create a separate keystore per app for better security and cleaner management. The effort is minimal — you do it once.
I forgot my keystore password. What do I do? ▼
If Play App Signing is enabled on Play Store (default for new apps): you can request a key reset from Google Play support. This is a one-time option per app lifetime. For apps on Amazon or distributed as APKs without Play App Signing — there is no recovery. You must create a new keystore and publish as a new app. This is why backing up the password matters as much as backing up the file.
What's the difference between keystore password and alias password? ▼
The keystore file is like a safe — the keystore password opens the safe. Inside the safe can be multiple keys, each identified by an alias. The alias password unlocks that specific key. For simplicity, most developers use the same password for both. Both passwords are required when signing a build.
Do I need a keystore to test on my own phone? ▼
No. When you export a Debug build in Godot, it auto-signs with a debug keystore that Godot manages. You can install and test debug builds freely. Only when you export a Release build — which you need for any distribution — does your release keystore get used.
Why does Firebase work in testing but not for Play Store users? ▼
Almost always a SHA mismatch. Your debug builds use the debug SHA, your release APK uses your upload key SHA, and Play Store delivers builds signed with Google's app signing key (which has a different SHA again). You need to register all three SHAs in Firebase: debug SHA, your upload key SHA, and the Play Console app signing key SHA. Go to Firebase Console → Project Settings → Your App → Add fingerprint.
Can I change my package name after publishing? ▼
No. The package name is permanent once an app is live on Play Store. Changing it creates a completely new app listing. Your existing users will not receive updates — they stay on the old version forever unless they manually install the new app. Set your package name correctly before first upload and never change it.
What keystore format should I use — .keystore or .jks? ▼
Both are functionally identical for signing Android apps. The .jks (Java KeyStore) format is the same thing with a different extension. Godot creates .keystore files by default. Either works — just be consistent with whichever you create. The content and cryptographic strength are the same.
My SHA-1 has colons but the service wants no colons. Which is right? ▼
Both are the same SHA fingerprint — one is colon-separated (AA:BB:CC) and one is continuous (AABBCC). The keytool command outputs the colon format. Firebase accepts both. AdMob typically wants no colons. Google Cloud Console for API keys also usually wants no colons. When in doubt, try without colons first — you can easily remove them by copying the fingerprint and replacing colons with nothing.
How do I find my keystore file location if I've forgotten where I saved it? ▼
Open Godot → Project → Export → Android preset. The Keystore/Release path field shows where Godot last looked for it. If the path is empty, check your export_credentials.cfg file in the project root (it may contain the path in plain text). If you used Android Studio, check your home folder. If all else fails, search your computer for files with .keystore or .jks extension.
